# SSC 검증랩 제작 표준 (Authoring Standard & Prompt)

> 이 문서는 새 `issue_type`의 검증랩(취약/조치 타깃 · 증적 · 가이드)을 만들 때 따라야 할
> **품질·일관성·정직성 기준**입니다. 사람 개발자용 체크리스트이자, 제작-시점 LLM에게
> 줄 수 있는 **프롬프트**로 그대로 사용합니다. (근거: 세션 내 반복 피드백을 codify)
>
> 사용법 — LLM 프롬프트로 쓸 때:
> *"너는 SSC 검증랩 템플릿을 제작한다. 아래 표준을 100% 준수하고, 5단계 요건과
> 검증 게이트를 모두 통과하는 초안을 만들어라. 통과 못 하는 항목은 만들지 말고 이유를 적어라."*

---

## 0. 3대 대원칙 (타협 불가)

1. **증적은 실제다 (Real).** 실제 취약 타깃을 세우고, 버튼 클릭 시 **실제 도구**(curl / openssl /
   dig / nmap / Playwright)를 실행해 **그 출력으로** 증적을 만든다.
   → 저장된 스크린샷 재사용, 과거 로그 재사용, 예상 결과 출력, 정적 HTML 결과 = **금지**.
2. **각 항목만 (Finding-scoped).** 소스 diff · 관측값 · 조치 전/후 · 조치 방향 — 모두 **그 finding에
   해당하는 것만**. 무관하거나 변하지 않는 항목을 함께 보여주면 오해를 부른다.
3. **정직하게 (Honest).** 항상 "참고용 PoC · 고객환경 아님 · 실제 해소는 SSC 재스캔". **실측**과
   **계산 판정**을 구분 표기한다.

> 핵심 구분: **미리 만들어도 되는 것 = 검증 환경(랩)** / **매번 새로 만들어야 하는 것 = 증적(실행 결과).**

## 1. 증적 파이프라인 (필수 흐름)

```
[사전·정적으로 준비]           [버튼 클릭 시 매번 실행]
- 취약 타깃(컨테이너/설정)      1. 컨테이너 기동/상태 확인
- 조치 타깃(컨테이너/설정)      2. 실제 점검 명령 실행 (취약 타깃)
- 점검 명령(도구)              3. 실제 점검 명령 실행 (조치 타깃)
- 정상/취약 판정 기준           4. 결과 수집 → 조치 전/후 비교
                              5. 실행 시각·대상·랩 버전 기록
                              6. 증적 화면 생성 (실행 결과로)
```

---

## 2. "취약점의 결과"를 보여줘라 (증적의 핵심)

조치 전/후는 **설정 유무**가 아니라 **취약점의 실제 결과**를 보여야 한다.

| | 조치 전(취약) | 조치 후(조치) |
|---|---|---|
| 원칙 | 취약점이 **실제로 발현**되는 결과 | 같은 방법으로 **해소**됨을 확인 |
| CSP | 삽입된 스크립트가 **실행**되어 쿠키 탈취 | 브라우저가 **차단**(실제 콘솔 위반 로그) |
| SPF | 위조 메일이 **수신 허용**(스푸핑 성공) | **격리·거부**(Received-SPF: fail) |
| 네트워크 | 포트 **open** | **closed/filtered** |
| 헤더 | Server/X-Powered-By **노출**, 보안헤더 **없음** | **숨김/적용** |

> "설정이 있다/없다"만 보여주면 *왜 위험한지*가 안 드러난다. 항상 **결과(impact)**를 보여라.

## 3. 카테고리별 증적 방식 (실제 도구 = 그 문제를 드러내는 방식)

| 카테고리 | 실제 명령/도구 | 증적 형태 |
|---|---|---|
| HTTP 헤더/토큰/쿠키 | `curl -sSI` | 터미널 응답 헤더 (해당 헤더만 강조, 나머지 dim) |
| CSP | 브라우저(Playwright) | 방명록 앱에서 스크립트 실행/차단 + **실제 콘솔 CSP 위반 로그** |
| TLS | `openssl s_client \| x509 -dates` | 터미널 인증서 실측 (해당 지표만) |
| DNS(SPF/DMARC/DKIM) | `dig +short TXT` | 터미널 레코드 + **수신 서버 위조 메일 판정** |
| 네트워크(포트/서비스) | `nmap -sT -Pn -p` | 터미널 포트 스캔 open/closed |

- **CSP만 브라우저 데모**인 이유: 결과가 화면(픽셀)에 보이는 유일한 케이스. 나머지는 헤더/포트/
  레코드라 픽셀에 안 보이므로 **명령 터미널**로 보여준다.
- 색 언어 통일: **빨강 = 위험/노출/취약**, **초록 = 안전/적용/차단**, 나머지(Date/ETag 등) dim.

## 4. 5단계 드로어 요건 (각 단계 "그 항목만")

순서: **① 개요 → ② 조치 방법 → ③ 조치 전/후 → ④ 관측값·확인 → ⑤ 마무리**

1. **개요** — 그 finding의 *무엇이 왜 문제인가*(why/심각도/분류). 그 이슈 것만.
2. **조치 방법**
   - *어디를 고쳐야 하나* = **여러 위치 중 택1**(모두 적용 아님)임을 명시.
   - **실제 소스 diff** = 랩 조치 타깃의 실제 설정에서 **이 이슈에 해당하는 줄만 발췌**.
     (랩이 여러 보안 헤더를 한 번에 적용해도, 전체 파일 diff 금지 → 오해.)
   - **엔진별 적용 탭**(헤더/TLS 계열): NGINX/Apache/IIS/App. "환경별 위치·문법 다를 수
     있음, 담당자 검토 후 적용" 명시.
3. **조치 전/후** — 그 취약점의 **실제 결과**를 실제 명령/재현으로(§2, §3).
4. **관측값·확인** — 그 finding의 **지표만**. (예: CSP = CSP 헤더 + 콘솔 증거 2행. Key Size·TLS
   Version 등 무관·불변 행 금지.) + 실측 검증 명령.
5. **마무리** — 그 이슈 **전용** 조치 방향(off-topic 단계 금지) + 고객 체크리스트 + **실행 로그(실제
   날짜·시간 UTC)**.

## 5. 새 issue_type 추가 절차 (체크리스트)

1. **실재 확인**: SSC `GET /metadata/issue-types`에 실제 존재하는 key인가? (v2 접미사 주의,
   유령 key 금지) — `node backend/scripts/labCoverageAudit.mjs` 로 대조.
2. **재현 가능성 판정**: 인프라 랩으로 재현 가능한가? (헤더/TLS/DNS/노출서비스=가능,
   유출/침해/패치지연/CVE=불가 → 가이드-only).
3. **카테고리 매핑**: `lab.js TEMPLATES` 의 http_header/tls/dns/network 중 어디인가.
4. **타깃**: 취약/조치 타깃이 이미 커버되나? (데이터 주도 우선 — 예: 네트워크는 포트 목록
   파라미터. 신규 컨테이너는 최후.)
5. **수집기 분기**: `evidence-collector`가 이 issue_type을 실제 명령으로 관측하도록 매핑.
6. **소스 diff**: `labSourceDiff.js` 에 이 이슈 발췌 규칙 추가(전체 파일 금지).
7. **관측값**: 이 이슈 지표만 반환하도록 `technical_diff` 구성.
8. **가이드**: `lab.js GUIDES` + `guideKey`에 이 이슈 전용 방향 추가(다른 이슈에 뭉뚱그리지 말 것).
9. **카탈로그**: `sandboxCatalog.js` 에 why/whereToChange/verification(대표 key + 필요 시 ALIASES).
10. **엔진 가이드**: 헤더/TLS면 `engineGuides.js shapeFor`에 매핑(엔진 무관이면 스킵).
11. **한글 라벨**: `KO_ISSUE_NAMES` / `KO_REMEDIATION`.
12. **검증 게이트 통과**(§6) 후 커밋.

## 6. 검증 게이트 (채택 전 반드시 전부 통과)

- [ ] **취약 타깃이 실제로 취약한가?** (도구로 취약 상태 확인 — open/none/실행됨)
- [ ] **조치 타깃이 실제로 고쳤는가?** (도구로 해소 확인 — closed/present/차단됨)
- [ ] **증적이 실제 명령 출력/실제 캡처인가?** (예상 결과·저장본 아님)
- [ ] **소스 diff가 이 이슈에 해당하는 줄만인가?** (전체 파일·타 헤더 섞임 없음)
- [ ] **관측값이 이 이슈 지표만인가?** (무관·불변 행 없음)
- [ ] **조치 방향이 이 이슈 전용인가?** (off-topic 단계 없음 · generic 폴백 아님)
- [ ] **실제 SSC key인가?** (`metadata/issue-types` 대조, 유령 아님)
- [ ] **정직 라벨이 있는가?** (참고용 PoC·고객환경 아님·SSC 재스캔, 실측/계산 구분)

## 7. 정직성 · 컴플라이언스 규칙

- 항상 표기: **참고용 PoC · 고객환경 검증 아님 · 실제 해소는 SSC 재스캔**.
- **랩 엔진(NGINX)을 고객 환경 조치 완료 증적처럼 제시 금지.** 랩은 "표준 환경에서 조치가 외부
  관측 결과를 개선함"을 증명할 뿐.
- **판정 기준은 설정파일이 아니라 외부 관측 결과**(HTTP 응답/TLS 협상/DNS 응답/포트) — SSC가
  outside-in 스캐너이므로 엔진 무관.
- 계산으로 낸 판정(예: SPF 위조 메일 판정)은 "실측 레코드 기반 수신 서버 판정"으로 **명시**하고
  실측(dig 출력)과 구분한다.

---

## 8. 엔진별 조치 가이드 규칙

- 조치는 **범용 목표값 × 엔진 출력 템플릿**으로 분해. (예: CSP 목표 =
  `Content-Security-Policy: default-src 'self'`, 엔진 = add_header/Header set/web.config/미들웨어.)
- 엔진 매칭이 필요한 건 **헤더/쿠키/TLS 계열뿐**. **DNS·네트워크는 엔진 무관**(탭 없음).
- **SSC는 웹 엔진을 신뢰성 있게 제공하지 않음(실측 확인).** → **수동 탭 선택이 기본**,
  `product_name` 감지 시에만 힌트(추정, 확인 후 적용).
- **버전**은 대체로 문법을 안 바꿈 → "버전 주의(caveat)"로만(주로 TLS: 1.3은 OpenSSL 1.1.1+).

---

## 9. LLM 활용 지침

- **런타임(버튼 클릭) 랩/증적 생성 금지.** 운영에는 LLM이 없고, 환각한 config/증적은 신뢰도·재현성·
  격리를 무너뜨린다.
- LLM은 **제작 시점(authoring-time)**에만: 새 타깃/가이드 **초안 생성** → §6 검증 게이트 자동/사람
  검토 → 커밋. (강한 모델 사용, 로컬 소형 모델은 코드 생성에 부적합.)
- 로컬 Ollama(exaone3.5)는 별개 기능(리스크/가이드 "해석")에만 사용. 랩 증적/가이드엔 미사용.

## 10. 금지 (Anti-patterns — 이번 세션에서 실제로 고친 것들)

- ❌ 예시 스니펫을 "실제 소스"처럼 제시 → ✅ 실제 파일 발췌
- ❌ 전체 파일 diff(타 헤더까지) on 단일 finding → ✅ 이 이슈 줄만
- ❌ 관측값에 무관·불변 행(Key Size, HTTP Status) → ✅ 이 이슈 지표만
- ❌ 조치 방향에 off-topic 단계(유효기간 이슈에 OCSP/CRL) → ✅ 이슈 전용
- ❌ 여러 이슈를 한 가이드로 뭉뚱그림(모든 인증서→"재발급") → ✅ 항목별 분리
- ❌ 유령 issue_type(SSC 미존재: x_powered_by_present 등) → ✅ metadata 대조
- ❌ 설정 유무만 보여줌(SPF 레코드 없음/있음) → ✅ 취약점 결과(위조 메일 통과/차단)
- ❌ 헤더 화면인데 모든 헤더 강조 → ✅ 이 finding 헤더만 강조, 나머지 dim
- ❌ 차단을 가짜 팝업/배너로 표현 → ✅ 실제 브라우저 콘솔 위반 로그
- ❌ NGINX 랩을 고객 조치 완료 증적처럼 제시 → ✅ 엔진 무관 관측 + 엔진별 적용 가이드
- ❌ 상대시간 로그([00:00]) → ✅ 실제 날짜·시간(UTC)
